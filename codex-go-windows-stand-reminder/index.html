Codex 帮我写了个 Go 版 Windows 久坐提醒工具：Stand Reminder

从想法到可用成品，一次和 Codex 协作完成的小工具实践

最近我做了一个挺实用的小工具：Stand Reminder。

它是一个运行在 Windows 端的久坐提醒程序，使用 Go 编写，常驻系统托盘，实测内存占用可以控制在 10MB 以下。项目本身不大，但很完整：从托盘常驻、状态检测、系统通知，到本地网页控制中心、统计视图、中英文切换，再到 GitHub Release 打包，已经是一套可以直接拿来用的成品。

这个项目还有一个对我来说比较有意思的点：它是我和 Codex 协作完成的。

不是那种“让 AI 写几行 demo”式的尝试，而是我负责提出需求、确认产品方向、不断补充使用细节，Codex 负责把这些想法快速落地，再一起迭代成一个真正能用的工具。

为什么会做这个工具

原因很简单：久坐太久了。

很多提醒类工具要么太重，要么交互别扭，要么打断感太强。我想要的其实很明确：

  • 平时安静地待在托盘里，不要打扰我
  • 真正连续使用电脑一段时间后，再提醒我起身活动
  • 如果我在摸鱼看视频，或者处于某种不适合打断的状态，识别最好能更合理一点
  • 最好还能有一个简洁的控制面板，随时查看状态、调参数、看统计
  • 占用尽量低，别为了提醒站起来反而多开一个“重量级软件”

于是就有了 Stand Reminder。

它现在能做什么

根据 README 里的功能说明和当前界面，Stand Reminder 目前已经具备这些核心能力：

  • Windows 托盘常驻运行，启动后不弹黑色终端窗口
  • 检测键盘、鼠标活动以及媒体播放状态
  • 按连续使用时长进行久坐提醒
  • 到达阈值后发送 Windows 系统通知
  • 点击通知后可直接打开控制中心
  • 提供本地网页控制中心，默认地址为 http://127.0.0.1:47831
  • 支持修改提醒分钟数、空闲重置分钟数、检测间隔、通知标题和通知内容
  • 支持手动 暂停、开始 / 恢复、我已起身活动
  • “我已起身活动”后会停止当前计时，并显示一个 10 分钟活动倒计时
  • 支持中英文界面切换
  • 支持今天、7 天、20 天三个维度的统计视图

如果只看功能列表，这已经不是一个“玩具项目”了，而是一个完整的桌面应用。

界面上，我比较满意的几个地方

从 README 展示图来看，这个项目的 UI 也不是“能用就行”的风格，而是认真做了组织和分层。

1. 控制中心首页

首页主要承担状态总览和操作入口的职责，打开后能快速看到当前运行情况，并执行常用动作。对这种托盘类应用来说，这样的设计很合理：

  • 日常不打扰
  • 需要时一键打开
  • 打开后信息集中，不用到处翻

2. 统计视图

统计页支持今天、7 天、20 天的视图切换，这一点我挺喜欢。

因为久坐提醒如果只有“提醒一下”，长期来看很容易被无视；但如果能看到自己的节奏和数据，它就更像一个轻量自我管理工具，而不仅仅是弹窗提醒器。

README 里也提到，今天统计图的 UI 思路参考了 Mizuki 的文章。这种做法我觉得挺好：尊重已有优秀设计，吸收思路，再结合自己的产品去实现。

3. 设置页

提醒时长、空闲重置时间、检测间隔、通知标题、通知内容都能自定义，说明这个工具一开始就不是“我写给自己能跑就行”，而是考虑到了不同人的使用习惯。

这也是我做工具时越来越在意的一点： 默认值重要，但允许用户调整同样重要。

从 commit 里能看出的迭代重点

这个仓库的提交记录也很有意思，能比较清楚地看出这个项目不是一次性写完的，而是持续在打磨。

我挑几个我觉得最能体现“从可用到好用”的 commit 来说。

1. feat: migrate storage to sqlite and refresh docs

这是一个很关键的升级。

从更简单的存储方式迁移到 SQLite，意味着项目开始认真处理统计数据和本地持久化能力。对于一个带历史统计、活动记录、状态管理的工具来说，SQLite 是很自然也很稳妥的选择。

这一步通常意味着项目从“逻辑跑通”进入“结构稳定、便于扩展”的阶段。

2. feat: improve stats dashboard visuals

我很认同这种迭代优先级。

很多开发者做完功能就停了，但真实使用体验往往卡在“看起来乱、不愿意打开”。统计面板视觉优化虽然不是硬功能，却会直接影响这个工具会不会被持续使用。

3. feat: open control center from notifications

这个细节非常加分。

提醒弹出后，如果用户还要自己去找托盘图标，再点开控制中心，流程就断了。现在直接从通知跳转控制中心，整个交互链条就顺了很多。这类改动不一定显眼，但非常体现产品感觉。

4. feat: 增加媒体播放的检测避免误判（大概有用吧）

我特别喜欢这个 commit 名，既诚实又有点好笑。

但它背后反映的是一个实际问题：“你在电脑前”不等于“你现在适合被提醒”。

如果正在放视频、开会、听内容，单纯靠输入事件判断，确实可能误判。加入媒体播放检测，说明这个工具在朝着“更像理解使用情境”而不是“机械计时器”的方向走。

5. feat: add PauseAfter configuration to engine and update tests

这属于典型的“后期好用度增强项”。

配置越来越细，意味着产品开始照顾不同节奏的人；同时还更新了 tests，也说明这个项目不是一味堆功能，而是在尽量保持可维护性。

6. 一系列 release / version / build 相关提交

比如：

  • ci: publish releases from version tags
  • ci: enrich release notes
  • fix: update version number to v0.2.3 in index.html and improve release bundle creation in build workflow
  • fix: update app version to v0.2.4 and remove version display from index.html

这些提交说明项目已经不只是“本地能跑”，而是在认真处理：

  • 版本发布
  • 自动化构建
  • Release 打包
  • 面向最终用户的分发体验

当一个项目开始反复打磨这些部分时，通常代表它真的准备给别人用了。

用 Codex 写这个项目，最大的感受

这次做下来，我对 Codex 的一个直观感受是：

它特别适合把脑子里的半成品需求，快速推进成一个结构完整的可运行项目。

尤其像这种桌面工具，需求会不断变化：

  • 托盘怎么做更自然
  • 通知如何触发
  • 页面结构怎么拆
  • 状态统计怎么存
  • 设置项应该暴露哪些
  • Release 如何打包

如果全靠自己一点点切换上下文，会比较消耗精力；而有了 Codex 之后，我更像是在持续做“产品经理 + 技术审稿人”的角色：

  • 我提出目标和细节
  • 它快速生成初版实现
  • 我再根据体验补充需求
  • 然后继续迭代

这种协作方式对独立开发小工具来说，效率真的很高。

当然，前提还是你自己要知道想要什么。Codex 能帮你加速，但不能替你判断什么才是合理的产品体验。

如果你想直接使用

如果你只是想直接试试这个工具，可以去 GitHub Releases 下载打包好的版本：

Stand Reminder Releases

下载 zip 后，运行里面的 stand-reminder.exe 即可。程序启动后会进入系统托盘。

如果你想自己编译，README 里给出的方式也很直接：

1

go build -ldflags='-H windowsgui' -o stand-reminder.exe .

这条参数里的 -H windowsgui 也正好对应了这个项目的目标： 它应该是一个安静待在后台的 Windows 工具，而不是一启动就冒出控制台窗口。

最后

我挺喜欢这种项目。

它不是什么宏大的系统，也不是为了炫技而做的复杂工程，而是一个很具体、很贴近日常使用场景的小工具：解决久坐问题，减少打扰，保持足够轻量，同时把体验做顺。

更重要的是，这个项目也让我更明确了一件事： AI 写代码真正有价值的地方，不只是“帮你补代码”，而是让一个原本可能会被搁置的小想法，更容易被做成、做完、做得像样。

如果你也在折腾 Go、小工具开发，或者也在尝试和 Codex 一起做项目，也许可以看看这个仓库：

https://github.com/sheetung/stand-Reminder
